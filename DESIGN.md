# mini-codex 设计文档

这份文档讲三件事:**它怎么运转、安全怎么分层、哪里故意没做以及为什么**。
比起"我实现了什么",这里更想讲清楚"我理解了什么"。

---

## 1. 一个 agent 到底是什么

最容易被误解的一点:**agent ≠ 会聊天的大模型**。

大模型本身碰不到文件系统、跑不了命令。它能做的只有一件事:**输出文字**。
所谓"调用工具",本质是模型输出了一段结构化的文字——"我想调 `read_file`,参数是 `main.py`"。
真正去读文件、把内容拿回来的,是**模型外面的代码**。

> Agent 的工作 = 把模型的"意图"变成"执行",再把执行结果变回对话,然后**再问一遍模型**,
> 直到模型不再提工具请求、给出最终答案。

这个"问 → 执行 → 回灌 → 再问"的循环,就是 agent。它在 [agent.py](agent.py)。

## 2. Agent loop(核心)

```
用户输入
   │
   ▼
┌──────────────────────────────────────────────┐
│  把 messages + 工具清单 发给模型                 │
│         │                                       │
│         ▼                                        │
│  模型返回"文字答案" 还是 "tool_calls"?          │
│    ┌────┴────┐                                  │
│    ▼         ▼                                   │
│ tool_calls  文字答案                              │
│    │         │                                   │
│    ▼         ▼                                   │
│ 过审批→执行  打印给用户 → 本轮结束 ✓              │
│ 结果塞回历史                                      │
│    │                                             │
│    └──► 回到顶部,带着结果再问模型 ◄──循环──       │
└──────────────────────────────────────────────┘
```

关键实现点:

- **MAX_STEPS = 15**:防止模型陷入"反复调工具"的死循环(对标失败模式里的 loop)。
  这是 "bounded blast radius" 在代码里最朴素的样子——给自主循环一个硬上限。
- **拒绝也要回灌给模型**:用户拒绝某次工具调用后,把"用户拒绝了"作为工具结果塞回历史,
  模型因此"知道"此路不通,会换个方案,而不是傻等或重复。

## 3. 工具系统:声明 / 路由 / 执行

工具有三件事要做([tools.py](tools.py)):

1. **声明**(`TOOLS_SCHEMA`)——用 OpenAI tools 格式写"说明书"给模型看。
   模型靠 `description` 决定**何时**调哪个工具。
   > 这段 description 是 AI PM 的核心交付物:写得烂,模型就不知道该调工具。
2. **路由**(`_DISPATCH`)——工具名 → 真正干活的函数。
3. **执行**——`_read_file` / `_write_file` / `_run_shell` 真正动手。

模型只产生"调用意图",执行权始终在我们代码手里——这是安全能介入的前提。

## 4. 安全:三层防御,以及它的诚实边界

### 实现了的三层

```
模型说"要执行 X"
   │
   ▼  第 1 层 黑名单:极危险命令直接拒,不问      (config.DANGEROUS_PATTERNS)
   ▼  第 2 层 审批闸门:写/跑操作执行前问人 y/n/a   (approval.py,代码强制)
   ▼  第 3 层 执行
```

第 2 层是**代码强制**的:不管模型问不问、怎么措辞,只要是 `write_file` / `run_shell`,
执行前一律停下等用户按键。这补上了"靠模型自觉"的不可靠。

### 诚实的边界(这部分比功能更重要)

做这套安全时,我亲手测出了它的洞,记录如下——**因为讲清楚边界,比假装没有边界更有价值。**

**洞 1:"模型问你确定吗" ≠ 安全机制。**
让 agent"删除所有文件",它会很懂事地回一句"确定要删吗?"。
但那是**模型自己客气**,不是我代码强制的。如果模型不问、直接执行,早期版本拦不住。
→ 教训:安全必须在代码里强制,不能依赖模型的善意。第 2 层就是为此而加。

**洞 2:黑名单是字符串匹配,必然有漏。**
黑名单拦 `rm -rf`,但 `Remove-Item -Recurse -Force *`(Windows 上等价)一开始就漏了;
补上 `remove-item` 后,`python -c "import shutil; shutil.rmtree('.')"` 照样不含任何关键词,
照样删光目录。**字符串黑名单永远列不全。**

**洞 3:审批是工具级,不是命令级。**
我的审批只判断"哪个工具"(`run_shell` 一律问),不判断"哪条命令"。
所以只读的 `dir` 也要问你(啰嗦),而危险命令换个写法又漏(不安全)。
真 Codex 用 exec policy **解析每条命令**,分成 自动放行 / 问人 / 拒绝 三档——粒度比我细。

**洞 4:"全部同意"(a)会让第 2 层闭嘴。**
一旦对 `run_shell` 按 `a`,审批就对所有命令放行,只剩会漏的黑名单在守。便利和安全的直接取舍。

### 为什么这些洞证明了 Codex 的四层防御是必要的

我们这三层(黑名单 + 审批 + 模型自觉)**没有一层能保证危险操作执行不了**。
这正是 Codex 还要再加两层的原因——而这两层我们刻意没做:

| 层 | 它补的死穴 |
|---|---|
| **沙箱**(第 3 层) | 不阻止命令,而是把它**锁在工作区、断网**。删了也炸不出去 = bounded blast radius 的终极形态。 |
| **Guardian**(第 4 层,AI 看 AI) | 用另一个模型**理解命令意图**,`shutil.rmtree` 这种语义删除,字符串看不出,Guardian 看得出。 |

> **结论:安全不是把某一层做到极致,而是承认每一层都会漏,用多层把漏的概率压到趋近于零。**

## 5. History 与 context 压缩(总结式压缩 + 保留首尾)

[history.py](history.py) 把整个会话的 messages 列表**收口到一个地方**:所有读写都过它。
正因为收口了,加 context 压缩时**只动了 history.py,agent loop 只多了一行**(调一下 `compress`)——
这验证了"好的预留让加功能不返工"。

压缩策略对标 Codex:历史(序列化后字符数)超阈值就触发,**保留头(system + 第一条任务)、
把中段总结成一段摘要、保留尾(最近 N 条)**。落地时守住了三个真实约束:

1. **总结要额外一次 LLM 调用**(不是纯数据操作)→ `compress(summarizer)` 把总结函数
   当参数注入,History 不绑定具体模型(`llm_client.summarize` 提供实现)。
2. **OpenAI 格式要求 tool_call 和它的结果严格配对** → 切尾部时,若起点是孤立的 tool 结果,
   就**往前挪到发起它的 assistant**,绝不拆散一组工具调用,否则下次调用 API 直接报错。
3. **摘要用 `user` 角色注入** → OpenAI 兼容接口对 user 消息位置最宽容,比在对话中段插
   system 消息更不容易报错。

> 设计心法:好的预留不是"把列表包一层",是"想清楚未来的具体实现,倒推现在该留哪些口子"。
> 这次压缩能一行接上,就是当初按这三个约束预留的回报。

## 6. 技术选型

- **直接用 `openai` 库,不用 LangChain**:agent loop 总共十几行,框架反而把核心机制藏起来,
  讲不清楚。手写一遍才知道每一步在干什么。
- **走 GLM 的 OpenAI 兼容接口**:改个 `base_url` 即可,工具调用走业界最通用的 `tools` 格式。
  选模型时看的不只是"能不能聊天",而是 **function calling 是否稳定**——这是 agent 的命门。

## 7. 与 Codex 的差距(一句话总览)

| 维度 | mini-codex | Codex |
|---|---|---|
| Agent loop | ✅ 有(~十几行) | ✅ 三层嵌套 + 流式 |
| 工具系统 | 3 个,工具级 | 几十个,命令级策略 |
| 安全 | 黑名单 + 审批(2 层) | + 沙箱 + Guardian(4 层) |
| 审批粒度 | 工具级 | 命令级(exec policy) |
| Context 压缩 | ✅ 总结式压缩 + 保留首尾 | 同款,另含更细的 token 估算 |
| 沙箱 | ❌ | OS 级(读/写/网三维) |

**差距是刻意的。** MVP 的目标不是复刻 Codex,而是把它的**核心机制讲到能跑、能讲清楚边界**。
